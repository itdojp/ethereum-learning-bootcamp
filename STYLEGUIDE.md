# STYLEGUIDE（教材/書籍）

このリポジトリの教材を「初心者が躓かずに読める書籍」へ育てるための、最小限の執筆ルール。

## 想定読者
- 初心者〜初級エンジニア（新卒〜実務経験2年未満）
- Ethereum / Solidity は初学者を含む

## 文体（トーン）
- 原則：**常体（だ・である調）**で統一する（ですます調は混在させない）
- 命令形は強くしすぎない（例：✗「必ず〜しろ」→ ○「まず〜する」「〜してから進める」）
- 不確かな点は断定しない（「要確認」「環境依存」などを明記する）

## 章の構造（おすすめ）
各章（Day）は、最低限この順序で揃えると読者が迷いにくい。
1. 学習目的（何ができるようになるか）
2. 前提（必要な環境/知識、事前に読む付録）
3. 理論解説（最小限→必要に応じて補足へ逃がす）
4. ハンズオン（コマンド単位で、期待される出力を添える）
5. つまずきポイント（原因→対策の順）
6. まとめ（3行で要約）
7. 提出物/チェックリスト（何を保存すれば再現できるか）
8. 実行例（`reports/DayXX.md` へのリンク）

## 用語・記号
- 初出では日本語で説明し、必要なら英語/略語を併記する（例：トランザクション（Tx））
- 同じ概念に複数表現を使わない（例：「コントラクト」「スマコン」を混在させない）
- 参照先を固定する：
  - 用語：`appendix/glossary.md`
  - つまずき：`appendix/verify.md` / `appendix/ci-github-actions.md` / `appendix/the-graph.md`

## コード/コマンド
- コマンドは `bash` ブロックで、コピペ可能にする（省略がある場合は「要置換」を明記）
- `PRIVATE_KEY` や API キー等は本文に直書きしない（`.env` と `.env.example` を使う）
- 出力例は「読者が比較できる最小限」にする（ハッシュやアドレスは `<TX_HASH>` 形式でもよい）

## リンク
- 可能な限り **相対パス**で貼る（GitHub 上でそのまま辿れる）
- 読む順序（目次）は `curriculum/TOC.md` を正とする
